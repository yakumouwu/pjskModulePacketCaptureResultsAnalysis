## 问题背景

一些 Unity IL2CPP 游戏会对 APK 内的 `libil2cpp.so` 或 metadata 做保护，导致：

- 直接用 APK 中原始 so 交给 `Il2CppDumper` 可能失败。
- 即使勉强通过，偏移也可能与运行态不一致，影响 IDA 分析。

文章的目标是：

- 从运行进程内存导出更“真实”的可执行映像。
- 让后续 Dumper / IDA 流程稳定可用。

## 核心思路

关键数据源：

- `/proc/<pid>/maps`：内存映射区间
- `/proc/<pid>/mem`：对应内存内容

基础判定逻辑：

1. 枚举目标进程相关映射段。
2. 读取段首 4 字节判断是否为 ELF（`7F 45 4C 46`）。
3. 对相邻段按规则合并，补齐可接受范围内 gap。
4. 输出候选 so/dump 文件，再用 Dumper 验证。

文章强调“不要只看 APK 原文件”，因为运行时映射通常更接近解密/重定位后的真实状态。

## 脚本方案

### 基础版流程

文中脚本大致包含：

- 通过包名拿 PID。
- 遍历 maps 区间并 dump。
- ELF 头判定输出扩展名（`.so` / `.dump`）。
- 对连续区做拼接，必要时补 gap。

### 增强版流程

后续迭代加入了：

1. 架构感知：同时匹配 `libil2cpp.so` 与 `split_config.<abi>.apk`。
2. 可追踪输出：文件名带包名与内存名，便于筛选。
3. 自动导出 maps：保留现场用于复盘。
4. `global-metadata.dat` 邻近提示：
   - 在 maps 中先记录 metadata 映射偏移。
   - 提示其后方最可能的目标 so 区间。
5. gap 保护阈值：
   - gap 过大时不盲目合并，避免污染候选文件。

## 实践结果

文中在至少两类游戏上验证：

- App Bundle 形态（拆分 APK）
- 非 App Bundle 形态

结果：

- 都能导出可被 `Il2CppDumper` 正常处理的候选文件。
- 与直接使用 APK 原始 so 相比，成功率显著提升。

作者也指出限制：

- dump so 仍可能需要后修复。
- 内存段边界后可能还有有价值数据未覆盖。
- 自动化只能减少试错，不能替代逆向分析本身。
