## Project SEKAI 逆向（1）：文件解密及 API 初步静态分析

### 1. 解密 Metadata

分析版本：ColorfulStage 2.4.1（Google Play 美服）。

作者先用 `Il2CppDumper` 直接处理 APK 中的 `global-metadata.dat` + `libil2cpp.so`，发现 metadata 受保护；随后从 `libil2cpp.so` 的 metadata 加载流程反推，定位到加载后对映射内容执行异或解密：

- 关键行为：`mapped_metadata[i] ^= METADATA_KEY[i & 0x7F]`
- 密钥长度：128 bytes 循环

因此可离线解密 metadata，再喂给 Dumper 继续分析。

文中还对比了 Unity 默认 `MetadataLoader::LoadMetadataFile`，指出默认流程不包含上述 XOR 逻辑，这一差异就是壳层所在。

### 2. 提取 libil2cpp.so

#### 准备

- 无 Root 实机场景下使用 WSA + Magisk。
- 从 `/proc/[pid]/maps` 获取运行时映射，借助工具导出内存中的 `libil2cpp.so`。
- 重点：优先取运行时映射（而非 APK 原始文件）。

导出后重新走 Dumper，`This file may be protected` 报错消失，说明运行时镜像更接近真实可分析态。

### 3. Dump 的部分发现

作者观察到业务命名层存在 BeeByte 混淆（以重命名混淆为主）：

- 类名有部分仍可读。
- 方法/字段混淆导致业务逻辑追踪成本较高。

### 4. API 解密，first look

- 抓包工具使用 Reqable。
- CA 信任可通过 Magisk 模块处理。
- 静态路径上可锁定疑似 API 加解密逻辑。

结论是后续应切到动态调试拿 Key/IV（第二篇展开）。

## Project SEKAI 逆向（2）：Frida 动态调试及取 API AES KEY/IV

### 0. 版本更新

从 2.4.1（美服）切到 2.6.1（台服）后，metadata 保护策略发生变化：

- 不再是早期全文件 XOR 模式。
- 原始 metadata 比旧版本多出额外头部数据（文中记录“多 8 bytes”现象）。
- 直接删头并不能完整恢复，字符串区仍有问题。

### 1. metadata 动态提取

作者复用运行时提取思路导出 metadata，并与 APK 静态文件对比：

- 运行时版本字符串区恢复可读。
- 再次 Dumper 后可顺利进入 IDA 级别分析。

### 2. frida-gadget 注入

思路是把 gadget 注入 APK 的运行时可加载 `.so`，重新签名安装。

文中记录了跨平台调试链路：

- Frida 跑在 WSL。
- ADB 后端在 Windows。
- 通过 `ADB_SERVER_SOCKET` 等方式串联。

注入验证工具：

- `frida-il2cpp-bridge`

### 3. IL2CPP 动态调用 runtime

定位 `Sekai.APIManager` 单例后，运行时读 `Crypt` 对象里的 `aesAlgo`，直接调用 `get_Key` / `get_IV`。

文中示例脚本核心逻辑：

1. 找 `Assembly-CSharp`。
2. 定位 `Sekai.APIManager`。
3. `gc.choose` 枚举实例。
4. 从 `get_Crypt()` 进入 `aesAlgo`。
5. 输出 key/iv 字节数组。

随后给出 Python `AES-CBC` 解密验证脚本，确认抓包 payload 可还原。

## Project SEKAI 逆向（3）：pjsk API 中间人攻击 POC

### 1. 工具

- `mitmproxy`
- 根证书注入（MagiskTrustUserCerts）
- 专用脚本：`mitmproxy_sekai_api.py`

### 2. 分析

文章修正了前文一个猜测：

- API 封包不是 protobuf，而是 MessagePack。
- 这意味着 schema 信息更贴近报文本体，调试时可直接解包转 JSON。

### 3. MITM

实验目标是修改乐曲难度相关字段：

- 响应中把 `musicDifficultyStatus` 改成 `available`，可点亮选项。
- 但服务端 Live 鉴权仍会拦截关键行为。

进一步尝试改请求中的 `musicDifficultyId`，可改变发包行为，但分数上报仍按服务端规则处理。

结论：

- 客户端字段篡改可影响 UI / 局部流程。
- 核心结算与权限仍受服务端约束。

### References

- <https://msgpack.org/index.html>

## Project SEKAI 逆向（4）：AssetBundle 反混淆 + PV 动画导入

### 1. 数据提取

项目采用热更新，包外资源体量大（数 GB 级）。

直接提取缓存文件看不到 `UnityFS` 魔数，说明有 AB 混淆层。

### 2. 加载流程分析

通过 dnSpy + IDA 追 `assetbundle` 相关类，最终定位到自定义流：

- `Sekai.AssetBundleStream`
- 内部根据 `_isInverted` 执行反混淆

关键处理模式：

- 跳过前 4 bytes
- 每 8 bytes block，对前 5 bytes 取反
- 仅头部窗口（典型前 128 bytes）需要处理

### 3. 提取资源

解混淆后可用 AssetStudio 正常读取。

文中记录当时 Unity 版本识别为 `2020.3.x`，资源可视化验证通过。

### 4. AssetBundleInfo

`AssetBundleInfo` 同样走与 API 相似的加密+封包流程，可用 `sssekai apidecrypt` 解出 JSON：

```bash
python -m sssekai apidecrypt .\AssetBundleInfo .\AssetBundleInfo.json
```

### 5. 资源使用

文章展示了从资源到 Blender / Unity 的落地链路：

- Blender 侧可检查 mesh 与绑定。
- Unity 侧可借 AssetRipper 回导工程。
- BlendShape 名称存在 CRC 映射问题，需要建立 name map 修复。

## Project SEKAI 逆向（5）：AssetBundle 脱机 + USM 提取

### 1. AssetBundleInfo

作者对比了设备缓存版与抓包全量版 `AssetBundleInfo`：

- 设备缓存约 4MB。
- 抓包全量约 13MB。

并指出新数据包含 `downloadPath` / `md5Hash` 等字段，可直接用于构造离线下载任务。

### 2. CDN

抓包可见 AB 下载统一挂在 CDN 根路径下，`downloadPath` 即相对路径。

通过二进制字符串也能反查到内置 CDN 地址线索。

### 3. 热更新 Cache

作者在 `sssekai/abcache` 实现了本地缓存管理，支持离线拉全量资源；
文中估算全量体积可到 `~27GB`。

### 4. 文件一览

离线资源中音视频占比高，拆包后可识别到 CRIWare 体系（USM/HCA）。

### 5. USM 提取

观察发现 USM 不是直接从 AB 裸取，而是经过缓存写盘流程：

- 运行后在 `/sdcard/Android/data/.../cache/movies` 出现可用文件。
- 使用 `WannaCRI` 可直接 demux。
- 结合 `MovieBundleBuildData` 可重建源流。

### References

- <https://github.com/mos9527/sssekai>
- <https://github.com/donmai-me/WannaCRI>

## Project SEKAI 逆向（6）：Live2D 资源

### 1. Live2D 模型

`[abcache]/live2d/` 下可找到模型和动画。

- `.moc3/.model3/.physics3` 可直接由 Live2D 官方工具打开。
- 贴图命名与映射信息可从 `BuildModelData` 补齐。

### 2. 动画 Key 预处理

动画在包内是 Unity `AnimationClip`，关键路径名以 CRC32 存储。

要导出到 `motion3`，必须先建立 `crc -> 参数名` 映射。

### 3. moc3 反序列化 + CRC 打表

文中给出基于 `moc3` 结构读取参数名/部件名的脚本，再统一生成 CRC 表：

- 读取 Header 与参数区偏移。
- 扫出 `Parameters/*` 与 `Parts/*` 字符串。
- 对完整路径做 `crc32`。

### 4. AnimationClip 转换

在 `sssekai` 中重现转换流程，可批量把 `AnimationClip` 转 `.motion3.json`。

示例命令：

```bash
sssekai live2dextract <abcache_path> <output_dir>
```

## Project SEKAI 逆向（7）：3D 模型

### 1. 文件结构

3D 资源主要位于 `[abcache]/live_pv/model/`，常见对象包括：

- Body（Skinned Mesh 主体）
- MonoBehaviour（附属数据，如碰撞相关）
- Texture2D（材质贴图）

### 2. 模型收集

导入侧的数据需求被拆为三块：

1. Static Mesh
2. Skinned Mesh + 骨骼层级 + 权重
3. BlendShape 与名称映射

其中骨骼与动画路径大量依赖 hash，需要反查路径字符串。

### 3. 模型导入

Blender 导入的关键工程点：

- Unity 左手系 -> Blender 右手系坐标变换
- 顶点数据在 flat float array 中按 stride 读取
- BlendShape 与 Vertex Group 名称长度限制
- 顶点色写入改用 Color Attributes（Blender 4.x）

### 4. Shaders

文中整理了主要纹理语义：

- `_C`：Base Color
- `_S`：Shadowed Color（阈值阴影）
- `_H`：高光/阈值相关通道
- Vertex Color：描边与高光强度控制

### 5. Shader 实现

实现路径为 NPR 常见组合：

1. 阈值 Diffuse（NdotL 分段）
2. Specular 混合
3. Emissive 叠加（基于 `_H` 通道）

### 6. 描边

采用 shell shading；并利用顶点色 `R` 通道控制局部是否描边，从而还原游戏中“同模型不同区域描边强弱不同”的效果。
